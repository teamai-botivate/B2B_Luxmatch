================================================================================
  SECTION 0 — WHAT THIS SYSTEM IS (ONE PARAGRAPH)
================================================================================

Jewel Factory is a B2B jewellery platform. A single MANUFACTURER maintains a
global design catalog (Gold only, no prices shown). Retail STORES register,
get approved by the manufacturer, and then browse + order from that catalog.
Each store has STORE MANAGERS who approve orders and custom design requests.
END CUSTOMERS walk into a store, use the in-store kiosk (no login) to browse
the manufacturer's catalog, do AR try-on, do similar-image search, place orders,
and submit custom design requests. Every customer order routes THROUGH the store
(a manager approves it) before reaching the manufacturer — and the manufacturer
NEVER sees customer personal data, only the store's identity + product specs.


================================================================================
  SECTION 1 — THE FOUR ACTORS
================================================================================

1. MANUFACTURER  (global admin — one company)
   - Login: /manufacturer/login  (cookie: mfr session)
   - Owns the global design catalog (manufacturer_products)
   - Approves/rejects store registrations
   - Receives B2B orders (store restock), kiosk orders (customer), and
     custom design orders — always sees STORE identity, never customer data
   - Uploads AR try-on transparent PNGs per design
   - Ships everything to the store's fixed delivery address
   - Portal: /manufacturer/*

2. STORE OWNER  (self-registers, one per store)
   - Register: /store/register  → status pending → manufacturer approves
   - Login: /store/login  (cookie: store session, carries jewellerId)
   - Full store dashboard: profile, branding, fixed delivery address
   - Add / remove Store Managers (owner-only)
   - Places B2B catalog orders (go to manager approval first)
   - Can do everything a manager can (owner acts as manager)
   - Portal: /store/*  (a.k.a. the /jeweller/* dashboard in old code)

3. STORE MANAGER  (multiple per store, added by owner)
   - Login: /store/manager/login  (cookie: manager session)
   - Same dashboard as owner EXCEPT: cannot add/remove managers,
     cannot change store settings
   - APPROVES: (a) customer kiosk orders, (b) custom design requests,
     (c) store B2B catalog orders — before any of them reach the manufacturer
   - Sees customer details (name, phone); manufacturer never does
   - Forgot/reset password via email
   - Portal: same as store owner

4. END CUSTOMER  (no login — in-store kiosk)
   - Visits the store device
   - Sees the MANUFACTURER's full catalog (not store inventory)
   - AR try-on, similar-image search — NO price shown anywhere
   - Adds to guest cart (sessionStorage), places kiosk order
   - Submits custom design requests (image + specs + name + phone)
   - All customer data stays with the store only
   - Kiosk pages: / , /catalog , /try-on , /search , /kiosk/custom-design ,
     /kiosk-checkout


================================================================================
  SECTION 2 — THE CORE PRODUCT RULES (never break these)
================================================================================

  - NO PRICE anywhere. Not in manufacturer form, not in catalog, not on any
    customer screen, not in any order. (Gold-only, price-on-request model.)
  - NO METAL field shown. Gold only. (Column can exist but stays nullable/hidden.)
  - AUTO DESIGN NUMBER: JF-0001, JF-0002 ... globally sequential, auto-generated
    when a manufacturer creates a product. This is the public product identifier.
  - CUSTOMER SEES MANUFACTURER CATALOG directly (not store's purchased inventory).
  - ALL CUSTOMER ORDERS ROUTE THROUGH THE STORE:
        customer order -> manager approves -> manufacturer
    Manufacturer sees: store name + store fixed address + product specs.
    Manufacturer NEVER sees: customer name, phone, or any customer detail.
  - STORE SELF-REGISTRATION -> manufacturer approval -> access granted.
  - STORE FIXED DELIVERY ADDRESS set at registration; manufacturer ships there.
  - BRANDING ON KIOSK: store's own logo + name (not "Jewel Factory").
    Footer credit: "Powered by AT Jewellers".
  - PORTAL/LOGIN pages: "Jewel Factory" branding.

================================================================================
  SECTION 4 — DATABASE DESIGN (PRISMA SCHEMA PLAN)
================================================================================

  Old system used raw Supabase SQL migrations + service-role key (RLS bypassed,
  filtering only in app code). NEW system uses Prisma against a fresh Supabase
  Postgres. Prisma gives typed models, migrations, and relations.

  IMPORTANT SIMPLIFICATIONS vs old schema:
    - MERGE `jewellers` + `stores` into ONE `Store` model.
      (Old code always auto-created a jewellers row per store and kept names in
       sync — a 1:1 coupling that added no value. One Store model = tenancy root.)
    - DROP legacy B-series tables entirely: customers, customer_addresses,
      customer_otps, cart_items, orders, order_items, order_status_history,
      branches, brands.
    - DROP PIN columns (pin_hash, pin_salt, idle_reset_*) from Store.
    - KEEP one tryon-assets model but make ownership explicit (belongs to a
      manufacturer product OR a store product — see below).
    - KEEP intelligence tables (ProductView, TryonEvent, ProductSale).
    - Add `trackingNumber` to KioskOrder explicitly (old guest_orders wrote it
      without a column — a latent bug).

================================================================================
  SECTION 5 — AUTH & COOKIES DESIGN (3 roles, clean)
================================================================================

  Old system had 5 cookies (lm_pin, lm_customer, lm_manufacturer, lm_store,
  lm_store_manager) with confusing shared secrets. NEW system: 3 cookies only.
  Customer has NO cookie (guest kiosk). PIN removed.

  Recommended: use one auth library approach consistently. Since it's a single
  Next.js app, you can use signed HTTP-only cookies (jose/iron-session) OR keep
  the old HMAC-SHA256 pattern (it works, is Edge-safe). Copy the working HMAC
  code from packages/tenant/src/index.ts if you want zero risk.

  COOKIE 1 — Manufacturer session
    payload: { manufacturerId }
    secret : MANUFACTURER_SECRET (own secret, min 32 chars)
    ttl    : 8h
    guard  : manufacturerGuard -> sets ctx.manufacturerId

  COOKIE 2 — Store session (owner)
    payload: { storeId }                 // storeId IS the tenant now (merged model)
    secret : STORE_SECRET
    ttl    : 8h
    guard  : storeGuard -> sets ctx.storeId, ctx.isOwner = true

  COOKIE 3 — Manager session
    payload: { managerId, storeId }
    secret : MANAGER_SECRET
    ttl    : 8h
    guard  : managerGuard -> sets ctx.storeId, ctx.managerId, ctx.isOwner = false

  KEY PATTERN TO KEEP (the good part of the old design):
    - Cookie carries storeId so tenancy resolves with ZERO db lookups.
    - managerGuard accepts BOTH the store cookie (owner) and manager cookie.
      -> owner can do anything a manager can. Set isOwner flag; branch where
         role matters (manager-management + store settings are owner-only).
    - Manager-management routes + store-settings routes use storeGuard (owner only).
    - Order/custom-design approval routes use managerGuard (owner OR manager).

  PASSWORDS: bcrypt (cost 10) for all three roles. ONE algorithm (drop scrypt).

  PASSWORD RESET (store owner + manager only):
    - password_reset_tokens: random 32-byte token, store SHA-256 hash, 1h expiry.
    - forgot-password always returns 200 (anti-enumeration).
    - Email the reset link via SMTP (nodemailer). Manufacturer has no self-reset;
      manufacturer resets STORE passwords, owner resets MANAGER passwords.

  TENANCY RULE (simpler now):
    - Every store-scoped query filters by storeId (from cookie, never from body).
    - Manufacturer catalog + custom/kiosk/b2b orders on the manufacturer side
      filter by manufacturerId.
    - With Prisma you CAN add real row-level checks in code cleanly (no service-
      role RLS-bypass footgun). Consider Prisma middleware that injects storeId.


================================================================================
  SECTION 6 — THE THREE ORDER FLOWS (step by step)
================================================================================

  ---- FLOW A: CUSTOMER KIOSK ORDER ----
  1. Customer on store kiosk browses manufacturer catalog (GET /api/kiosk/catalog).
  2. Adds items to guest cart (sessionStorage 'guest_cart'). No login.
  3. Fills checkout form: name, phone, email?, pickup/delivery, address?, notes.
  4. POST /api/kiosk/orders
       - resolve storeId (kiosk device is tied to a store)
       - resolve store + manufacturerId
       - create KioskOrder (pendingStoreApproval=true, forwardedToManufacturer=false)
       - create KioskOrderItems (snapshot product name/image/category)
       - create first status history row
  5. Manager sees it in "Pending Approvals" (GET /api/manager/kiosk-orders/pending).
  6. Manager APPROVES -> sets storeApprovedBy/At + forwardedToManufacturer=true.
     Manager REJECTS  -> status CANCELLED.
  7. Manufacturer sees approved kiosk orders (GET /api/manufacturer/kiosk-orders)
     -> shows STORE NAME prominently, NEVER customer data.
  8. Manufacturer advances status PENDING->CONFIRMED->PACKED->SHIPPED->DELIVERED.
  9. Store tracks status; hands jewellery to customer.

  ---- FLOW B: STORE B2B ORDER (restock) ----
  1. Store owner browses manufacturer catalog (GET /api/store/catalog).
  2. Adds to B2B cart (sessionStorage 'b2b_cart').
  3. Reviews cart + delivery address (auto-filled from store fixed address) + notes.
  4. POST /api/store/orders
       - create B2bOrder (pendingManagerApproval=true)
       - create B2bOrderItems, first history row
  5. Manager sees pending B2B orders -> APPROVE / REJECT.
  6. On approve -> reaches manufacturer.
  7. Manufacturer advances status. On DELIVERED -> fulfillB2BOrder():
       - copy each manufacturer_product into the store's Product table
       - copy images + tryon assets
       - set stockCount = quantity (or increment if already exists)
       - record fulfilledAt + fulfilledProductIds
  8. Store now has that design in its own inventory (for intelligence/tracking).

  ---- FLOW C: CUSTOM DESIGN REQUEST ----
  1. Customer on kiosk fills /kiosk/custom-design:
       image (upload or URL) + category + weight + purity(dropdown) + notes
       + name + phone.
  2. POST /api/kiosk/custom-design -> create CustomDesignRequest (status PENDING,
     WITH customer PII).
  3. Manager sees it (GET /api/manager/custom-designs) with customer details.
  4. Manager APPROVE -> forwardCustomDesignToManufacturer():
       - create CustomDesignOrder = SANITIZED copy:
           store name + store fixed address + category + weight + purity +
           reference image + design notes. NO customer name/phone.
       - request.status = FORWARDED
     Manager REJECT -> request.status = REJECTED.
  5. Manufacturer sees CustomDesignOrder (GET /api/manufacturer/custom-designs)
     -> store identity + specs only.
  6. Manufacturer advances status PENDING->CONFIRMED->IN_PRODUCTION->...->DELIVERED.
  7. Manager portal JOINS request->order to SHOW manufacturer status live
     (this was the recent bug fix — keep the join).

  [C1] KIOSK -> STORE RESOLUTION  (DECIDED: URL-PATH)
     Customer has no login. Every kiosk request must know which store it is.
     MODEL: URL path. Each store's kiosk lives under /<storeSlug>/...
            e.g. jewelfactory.com/abc/catalog , /abc/try-on , /abc/kiosk-checkout.

       segment for ALL customer/kiosk pages.

         app/[storeSlug]/kiosk/custom-design/page.tsx
         app/[storeSlug]/kiosk-checkout/page.tsx  (+ /success)
         app/[storeSlug]/compare , /saved
       Each page reads params.storeSlug, resolves the Store (cache the lookup),
       404s if the slug is unknown or the store is not active/approved.
     - APIs: kiosk API calls carry the storeSlug (path segment or a header the
       client sets from the URL). Resolve storeId server-side from the slug;
       NEVER trust storeId from the request body. Options:
         (a) /api/kiosk/[storeSlug]/orders  (slug in the API path), or
         (b) client sends X-Store-Slug header taken from window URL, guard
             resolves + validates it.  Pick one and be consistent.
     - Store branding (logo/name/city) on the kiosk = the resolved store's fields.
     - Portal/staff routes (/portal, /store/*, /manufacturer/*) are NOT under
       [storeSlug] — they stay at the root.
     - Local dev: just visit /abc/... — no DNS setup needed (advantage over
       subdomains). A default/demo slug can be seeded for quick testing.
     NOTE: this replaces the old lm_store-cookie / SHOP_JEWELLER_ID kiosk hack.
     WHY URL-path over subdomain: no wildcard DNS, no wildcard TLS, works on
     localhost and any host immediately, simpler deploy.

  [C2] SEED / FIRST-RUN  (DECIDED: pre-seeded, same as current live system)
     - Manufacturer has NO signup UI. Seed exactly one manufacturer row
       (name, email, bcrypt password) via a prisma seed script.
     - Seed the Category list (Ring, Earring, Necklace, Bangle, Pendant, ...).
     - Optionally seed a demo store (approved) for testing.
     - prisma/seed.ts run via `prisma db seed`. Document the manufacturer login
       credentials in a secure place (not in the repo).

  [C3] ORDER NUMBER — no collisions  (ALREADY FIXED in current live system)
     - Old: 4-digit Math.random() -> only 9000/day -> same-second collision on
       the UNIQUE order_number constraint -> insert fails.
     - Fixed: high-entropy suffix (epoch-ms + base36 random) on GK-/B2B-/CD-.
     - In the NEW Prisma system, prefer a DB sequence or a dedicated counter
       table per prefix per day, OR keep the high-entropy string. Either is fine;
       just don't go back to a small random range.

  [C4] TRANSACTION SAFETY  (partially fixed; finish in rebuild)
     - Multi-insert flows must be atomic or they leave half-written data:
         * fulfillB2BOrder  (inserts products + images + tryon assets + updates)
         * placeGuestOrder / placeB2BOrder (order + items + history)
         * forwardCustomDesignToManufacturer (order insert + request update)
     - supabase-js can't do cross-statement transactions -> old code couldn't.
       Prisma CAN: wrap each of these in `prisma.$transaction([...])` (or the
       interactive `prisma.$transaction(async (tx) => {...})`).
     - forwardCustomDesignToManufacturer already got a manual rollback in the
       live system (delete the inserted order if the request update fails);
       in Prisma this becomes a single transaction — cleaner, do it there.

  [C5] FOREIGN-KEY OWNERSHIP OF ORDER ITEMS  (root cause of two live bugs)
     Two live bugs came from FK mismatches — design the schema to avoid them:
       * guest_order_items.product_id FK -> store `products`, but customers order
         from the MANUFACTURER catalog. Fix was product_id = null + snapshots.
         REBUILD: KioskOrderItem.manufacturerProductId (FK -> ManufacturerProduct)
         is the correct link, NOT a store-product FK. Keep snapshots too.
       * reviewed_by / *_approved_by FK -> store_managers, but the OWNER (not a
         manager) also approves; owner's storeId isn't a manager row -> FK fail
         -> the status update silently failed -> duplicate orders.
         REBUILD: make these columns nullable and store null for owner-approved,
         OR add an `approvedByRole` enum + nullable managerId. Never write a
         non-manager id into a store_managers FK.
